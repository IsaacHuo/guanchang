1. 宏观视角：数据大屏与统计看板 (Dashboard)
普通用户刚进网站，需要直观感受到数据的宏大和趋势。
* 中国地图热力图 (Geo-Map)：
    * 展示方式： 一张中国地图，各省份颜色的深浅代表该省落马官员的数量或级别权重。
    * 交互： 悬浮显示具体数字，点击省份下钻到该省的落马官员列表或关系网。
* 时间轴与趋势图 (Timeline & Line Chart)：
    * 展示方式： 柱状图或折线图展示每年/每月落马人数趋势。可以标注重要历史节点（如十八大、十九大、某次重要会议）。
* 多维分类统计 (Pie/Bar Charts)：
    * 按级别分类（副国级、省部级、厅局级、县处级等）。
    * 按系统分类（政法系统、金融系统、医疗系统、国企等）。
    * 按处分结果分类（双开、判处死缓、无期徒刑等）。
2. 中观视角：核心卖点——关系图谱 (Graph Visualization)
这是 Neo4j 数据库的绝对强项，也是你网站的灵魂。必须重点设计。
* 窝案/派系可视化 (Cluster Graph)：
    * 展示方式： 将官员（Node）、工作过的部门（Node）用关系线（Edge）连接起来。例如展示“山西塌方性腐败”、“某某系统贪腐案”。
    * 节点设计： 节点大小代表官员级别，颜色代表状态（调查中、已判决）。
    * 连线设计： 连线代表关系，如“上下级”、“同僚”、“亲属”、“行贿/受贿”。
* 路径发现 (Path Finding)：
    * 交互： 用户输入两个官员的名字，前端调用 Neo4j 的最短路径算法，展示这两个人之间是通过哪些人或机构联系起来的。
* 动态展开 (Expand Node)：
    * 交互： 不要一开始把所有数据扔到图上（会变成一团乱麻）。从一个核心人物（如某个大老虎）开始，用户双击他的头像，像水波纹一样展开他的“秘书”、“亲属”、“提拔的下属”。
3. 微观视角：官员个人数字档案 (Personal Profile)
当用户对某一个具体官员感兴趣时，提供详细的个人页面。
* 履历与落马时间线 (Career Timeline)：
    * 左侧是一条垂直的时间轴，记录他的升迁轨迹；右侧用红色高亮标出他在哪个阶段开始违纪违法。这能直观看出“带病提拔”或“长期潜伏”的特征。
* 个人专属关系网 (Ego-Graph)：
    * 在这个人的主页，只展示以他为中心的一度或二度关系图谱。
* 关键标签 (Tags)：
    * 用类似词云的形式提炼官方通报中的高频词，如“搞团团伙伙”、“权色交易”、“对抗组织审查”、“敛财过亿”等。
4. 工具属性：强大的检索系统 (Search & Filter)
* 多条件组合筛选： 提供类似电商网站的筛选栏，允许用户按名字、籍贯、落马时职务级别、落马年份、所属系统进行交叉检索。
* 全文检索： 支持对官方通报全文进行检索，找出包含特定关键词的所有官员。

🛠️ 前端技术栈与可视化库推荐
既然后端是 Neo4j，前端需要强大的图可视化能力。
1. 图可视化库（最重要）：
* AntV G6 (强烈推荐)： 蚂蚁金服开源的图可视化引擎，专门为关系图谱设计。交互极其丰富（缩放、拖拽、节点展开收起），并且 UI 默认就很现代、美观。
2. 整体前端框架：
* Vue 3 / React： 目前的主流选择。如果你想做单页应用（SPA）并保持复杂状态（比如图谱筛选器），这两个框架必选其一。
* Tailwind CSS： 用于快速构建冷峻、严肃风格的 UI 界面。

🎨 UI/UX 视觉风格建议
* 主色调建议：
    * 档案风 (Archive Style)： 以泛黄的纸张颜色或冷灰色为主，配合类似盖章的 UI 元素（如红色的“查”、“双开”印章效果），营造历史档案的感觉。
* 避免过度娱乐化： 题材严肃，切忌使用过于花哨的动画或卡通色彩，应该呈现一种客观、数据驱动的“智库”感觉。

加入一个关系型数据库PostgreSQL：

通常采用的架构是：PostgreSQL（主库） + Neo4j（图计算辅助库）。
实现：
1. 大规模列表与分页展示： 你的网站肯定有一个页面叫“落马官员名册”，需要按年份、级别、省份进行筛选并翻页（比如每页 20 条）。Neo4j 做这种传统的分页列表（ORDER BY ... LIMIT ... OFFSET）效率不如 MySQL。
2. 存储长文本： 官方纪委的通报往往是一长段文字（比如“丧失理想信念，背弃初心使命...”）。把几千字的长文本存成 Neo4j 节点的一个属性是非常低效的，这会让图数据库变得臃肿。长文本应该存在 MySQL 里。
3. 复杂统计图表： 如果你想画一个柱状图（比如“2012-2023年各省落马人数对比”），这需要做大量的 GROUP BY 和聚合计算，这是关系型数据库（或数据仓库）的强项。
4. 常规业务数据： 网站的用户注册、登录、访问日志、评论等，肯定要存在关系型数据库里。


💡 建议的最终架构与分工：
1.  PostgreSQL (作为数据基座 - Source of Truth)
    * 存储官员的基本信息（ID, 姓名, 性别, 籍贯, 职务序列, 通报长文本, 照片链接）。
    * 负责网站的：列表页、分页、大盘统计图表、按条件筛选。
2. Neo4j (作为关系引擎 - Graph Engine)
    * 只存储轻量级数据！ 节点只存：官员ID、姓名、级别。连线只存：关系类型（如“提拔”、“亲属”、“行贿”）。
    * 负责网站的：关系图谱展示、发现两个官员之间的隐藏关系链、查找某人的派系圈子。
